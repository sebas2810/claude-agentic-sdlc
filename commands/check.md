You **drain your queue**: read the board **once**, then handle every piece of work eligible for this seat's role in that one snapshot — take it, report, take the next-from-the-same-snapshot — repeating (item → report → next) until none remain for your role, then report `queue clear — idle` and **stop**. Two rails: the drain is **operator-initiated** (this `/check`) and **bounded by the work that exists right now**, and every item still passes its normal gate. **Stop at empty:** once your queue is clear you do **not** keep re-reading the board (no self-loop, no idle-poll) — the operator runs `/check` again when new work lands. The whole drain costs **one** expensive board read regardless of queue depth; after that, only cheap per-item operations (your own item's status-field mutation + PR/REST ops) — this is what preserves the rate-limit fix.

Resolve once:
```
BOARD_ID="${BOARD_ID:?BOARD_ID unset — run /check in a configured seat pane (set BOARD_ID/BOARD_OWNER in .env.local)}"
BOARD_OWNER="${BOARD_OWNER:?BOARD_OWNER unset}"
ROLE="${SEAT_ROLE:?run /check inside a seat pane}"; KEY="${SEAT_KEY:-$ROLE}"
```
Read the board **once** per engagement: `gh project item-list "$BOARD_ID" --owner "$BOARD_OWNER" --format json --limit 300` (a deliberate, operator-initiated read — not a self-paced poll). This single 300-item read is the **expensive** GraphQL call; you **drain from this one snapshot** and do **not** re-read the full board after each item — each item costs only cheap single-item operations (its own `Status`-field mutation + PR/REST ops). That keeps the drain at one board read regardless of how many items you clear (the rate-limit fix the operator-driven model exists for).

Then act by **role**:

- **producer** (`engineer`; KEY = your `seat:<x>` suffix): the next item with `Status=Scoped`, label `seat:$KEY`, **no assignee**. → CLAIM it (flip `Scoped→In Progress` + assign yourself), read its issue + its `## Steer` AC (a **re-`Scoped`** item also carries QA's per-criterion comments — address them), build per your KICKOFF (branch off `origin/main`, gates + a real deployed round-trip), open ONE PR with `## Closes #<n>`, set `Status→Delivered`, post your ready-signal. **Never self-merge.** **Block protocol:** if you hit a genuine **consult-exception** — the AC can't be met as written, a real product fork, or out-of-scope creep — do **not** build. Post the **full consult-exception to the GitHub issue** (file-cited findings · the fork/options · your recommendation), set `Status→Blocked` (+ assign yourself so it's owned), then **stop**. The issue comment IS the board item's context — the SM/PM read it from the board, not your pane.

- **quality-engineer**: the next item with `Status=Delivered`. → VERIFY it against its pre-committed AC on the **deployed** env (perturb the happy path — gate reliability, not one lucky output). Post per-criterion PASS/FAIL; on PASS set `Status→Tested`, on **FAIL set `Status→Scoped`** with the per-criterion comments (the engineer re-pulls it next `/check`). **You never merge.**

- **scrum-master**: the next `Status=Tested` item. → Validate the merge preconditions — a **real QA PASS** verdict, **CI green**, the PR **mergeable / clean** — then **MERGE it (squash; 4-eye — you did not author it)** and drive `Merged→Released` (staging deploy + canary; PROD stays owner-gated). If a precondition fails, **route, never force-merge**: dirty/conflicting PR → back to the engineer to rebase; no QA verdict → back to QA. Plus board hygiene — explode any newly-framed Epic into sub-issues (back-link the `#`s), enforce WIP, sweep aging/`Blocked`. Two `Blocked`-specific duties: **(a) operationalize PM re-frames** — flip `Blocked→Scoped` for any item the PM has re-framed/approved (the PM posts the decision; *you* perform the status transition, then the producer pulls it); **(b) verify before surfacing** — for each `Blocked` consult-exception, **independently verify its claims** (sanity-check the cited findings against the codebase/board) and surface to the PM **with a verdict** — *legit blocker* / *avoidable* / *needs-PM-product-call* — never a bare relay. (Producers pull their own `Scoped` via `/check`, so you don't push build work.)

- **pm**: **no routine merge.** Frame the top `Backlog` item → `Scoped` with a **falsifiable** AC (the contract QA verifies against). Resolve any product/scope judgement the QA seat surfaced (an ambiguous or deploy-gated AC), or re-frame a `Blocked` consult-exception the SM surfaced (e.g. trim the AC + "approved → Scoped"), so the SM can move it. **You frame/decide by posting a comment — you do NOT flip the board `Status` field.** The SM operationalizes the transition (e.g. flips `Blocked→Scoped` after your re-frame). Act only on a real product/scope item; **no status edits**. Otherwise oversight — roadmap + owner touchpoints.

If there's nothing for your role — the first pull, or any later pull during the drain, finds no eligible item in the snapshot: report `queue clear — idle` (or `no <status> work for <seat> — idle`) and stop; do **not** keep re-reading the board.

**Drain your queue, then idle.** Don't stop after one item: from the single board snapshot above, pull → do → report (with the issue/PR #) → **pull your role's next eligible item from that same snapshot** → repeat (item → report → next) until none remain, then report `queue clear — idle` and **stop**. This wraps **every** role above — a producer drains its `Scoped` lane, the quality-engineer its `Delivered` queue, the SM its `Tested` + flow queue, the PM its framing/judgment queue. Two rails hold: **(1)** the drain is **operator-initiated** (this `/check`) and **bounded by the work that exists right now** — every item still passes its normal gate (producers stay Engineer → QA → SM per unit; this is **not** autonomous EPIC-draining); **(2) stop at empty — no idle-poll:** once your queue is clear, do **not** keep re-reading the board (no self-loop, no `/loop`, no `ScheduleWakeup`) — the operator re-engages you when new work lands. Cost stays at **one** board read per engagement (cheap per-item mutations only after).
